## ZMQ RPC 框架说明

在 ZMQ RPC 框架中：
- **RPC Client** = 发起请求的一方（通常在 Worker 上）
- **RPC Server** = 处理请求的一方（另一个 Worker 或 Master）

### Framework 各字段含义

| 字段 | 含义 | 执行位置 |
|------|------|---------|
| client_req_framework_us | RPC Client 准备请求的时间 | RPC Client 端 |
| server_req_queue_us | 请求在 RPC Server 队列中等待的时间 | RPC Server 端 |
| server_exec_us | RPC Server 实际处理时间 | RPC Server 端 |
| server_rsp_queue_us | 响应在 RPC Server 出队队列中等待的时间 | RPC Server 端 |
| client_rsp_framework_us | RPC Client 接收响应的时间 | RPC Client 端 |
| network_residual_us | 网络传输时间 (双向) | 网络层 |

### RPC 调用流程

```
RPC Client                         RPC Server
   │                                  │
   │── client_req_framework (准备) ──>│ server_req_queue (排队)
   │                                  │ server_exec (处理)
   │                                  │ server_rsp_queue (响应排队)
   │<── client_rsp_framework (接收) ──│
   │
   │======== network_residual (网络) =====│
```

## 执行摘要

### 主要发现

1. **网络传输主导 E2E 延迟 (99%+)**
   - `network_residual_us` 占端到端延迟的 99.4-99.8%
   - URMA/RDMA 传输是主要瓶颈

2. **URMA 连接建立开销 (~8ms/新连接)**
   - 完整连接建立需要 7-8ms
   - Jetty 创建、段导入、连接握手

3. **特定 RPC Server 请求队列瓶颈**
   - RPC Server 192.168.42.114 显示高 server_req_queue (3068us)
   - 尽管有空闲线程，请求仍然排队

4. **特定路径 RPC Server 响应队列问题**
   - 路径 192.168.45.216 -> 192.168.219.66 显示高 server_rsp_queue (4084us+)
   - 表明出方向网络拥塞

5. **URMA Infight Write Count 波动**
   - 范围: 1-10
   - 值为 10 时可能表示拥塞

## 分类问题发现

### 1. client_req_framework (min: 662us, max: 3668us)

**发现**: RPC Client 端框架开销很小。大部分延迟在后续阶段。

**根因**: RPC Client 准备请求极快，优化收益低。

---

### 2. e2e (min: 41037us, max: 83567us)

**发现**: E2E 延迟是**网络 bound**。应聚焦 RDMA/URMA 优化。

**URMA 连接模式**:
| 连接类型 | 开销 |
|---------|------|
| 新连接 | ~8ms |
| 复用连接 | 极小 |

---

### 3. framework_breakdown (min: 40972us, max: 83486us)

**发现**: RPC Server 端处理 (server_req_queue + server_exec + server_rsp_queue) < 200us 合计。

**RPC Server 处理分解**:
- server_req_queue: 65-97us (正常)
- server_exec: 42-58us (极快)
- server_rsp_queue: 18-24us (正常)

**结论**: RPC Server 处理极快，无瓶颈。

---

### 4. network_residual (min: 40664us, max: 83252us)

**发现**: 网络传输是延迟的根本原因。

**URMA 详情**:
- 内存区域: 每个连接 12GB registered
- NUMA 亲和性: 全部使用 `useNumaAffinity:1` (最优)

---

### 5. server_exec (min: 1527us, max: 3685us)

**发现**: RPC Server 执行时间有变化; eviction manager 增加延迟。

**高 Exec Trace (0d34feb8)**:
- server_exec: 3654us
- URMA wait: 3.57ms

**高队列 Trace (1051ba83)**:
- server_req_queue: 3068us
- Eviction list: 1226 项

---

### 6. server_req_queue (min: 585us, max: 3149us)

**发现**: RPC Server 192.168.42.114 显示显著的请求排队延迟。

**根本原因**:
- 线程池有空闲线程但请求仍排队
- 可能存在线程亲和性或调度问题

---

### 7. server_rsp_queue (min: 4084us, max: 8810us)

**发现**: 路径 192.168.45.216 -> 192.168.219.66 有高响应队列延迟。

**分解**:
- server_req_queue: 17-19us (低)
- server_exec: 320us (低)
- **server_rsp_queue: 4084us (89%+) (非常高)**

**根本原因**: RPC Server 响应发送是瓶颈，暗示出方向网络拥塞。

## 建议总结

| 优先级 | 问题 | 建议 |
|--------|------|------|
| P0 | 网络延迟 99%+ | Profile RDMA 栈，检查网络拓扑 |
| P0 | URMA 连接开销 | 实现连接池/缓存 |
| P1 | RPC Server 192.168.42.114 高队列 | 调查线程亲和性，增加线程 |
| P1 | 路径 45.216->219.66 高响应队列 | 检查网络路径拥塞 |
| P2 | 高 urma_inflight_wr_count | 监控阈值，增加 QP 深度 |

---

## 结论

Traces 揭示了一个**网络 bound 系统**，其中分布式对象缓存操作由 RDMA/URMA 传输时间主导 (99%+)。RPC Client 和 RPC Server 端处理都极快 (<200us)，优化 RPC 代码收益极低。网络路径和连接建立开销是主要优化目标。

**后续步骤**:
1. 在受影响的网络路径上 Profile RDMA 性能
2. 实现 URMA 连接池化
3. 调查 RPC Server 192.168.42.114 的线程调度
4. 检查 worker 对之间的网络拓扑
