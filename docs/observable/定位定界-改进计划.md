# 定位定界体系改进计划（Plan）

本文基于当前 `docs/observable` 现状，给出下一阶段可落地的整改方向，目标是让系统做到：
- 更早告警（分钟级发现）
- 更快定界（责任层级 + 模块级）
- 更可复盘（同一请求跨端证据可重建）

---

## 1. 当前主要短板（按优先级）

1. **告警以“结果失败”居多，缺少“前置信号”**
   - 现在主要看 `success_rate` / `p99` / 错误码突增。
   - 对链路恶化（重传、队列堆积、UB fallback 激增）预警不足，常在业务已超时后才触发。

2. **错误码语义混叠，定位成本高**
   - `K_RPC_UNAVAILABLE(1002)` 同时覆盖网络不可达、对端未监听、服务内部不可用，外部看起来像一个桶。
   - `K_RUNTIME_ERROR(5)` 在多模块被复用，边界不清，难以直接定责。

3. **分段时延证据不连续**
   - 已有“①~⑥分段”方法论，但缺乏统一结构化日志字段和统一 trace 键，难自动聚合出阶段瓶颈。

4. **跨端关联能力弱**
   - SDK/Worker/依赖侧日志可对齐，但没有强制字段契约（trace_id/request_id/worker_instance_id/stage）。

---

## 2. 告警体系整改（从“事后告警”到“分层告警”）

## A. 告警分层

- **SLO 层（业务体验）**
  - `success_rate_1m`、`read_p99_1m`、`write_p99_1m`
  - 触发：快速判断是否用户可感知

- **故障类型层（错误码桶）**
  - TCP/RPC：`1000/1001/1002/19`
  - UB/URMA：`1004/1006/1008`
  - 控制面：`14/25/31/32`
  - 资源：`6/K_NO_SPACE/K_IO_ERROR`

- **前置信号层（预测性）**
  - `ub_fallback_rate`
  - `retry_count_p95`
  - `worker_queue_depth`
  - `etcd_watch_lag_ms`
  - `socket_reset_rate`

## B. 告警规则建议（第一版）

- `P1`：`success_rate_1m < 99.0%` 且持续 3 分钟
- `P1`：`read_p99_1m` 或 `write_p99_1m` 超基线 2 倍且持续 5 分钟
- `P2`：`1002` 占比 > 30% 且入口 Worker 无对应请求日志（偏 L1（平台与网络））
- `P2`：`1006` 占比 > 20% 且 `ub_fallback_rate` 同时升高（UB 链路风险）
- `P2`：`K_SCALING(32)` 持续 > 10 分钟（迁移不收敛）
- `P2`：`K_OUT_OF_MEMORY(6)` 连续增长 + `allocator_fail_count` 增长

## C. 告警输出格式（必须统一）

每条告警至少包含：
- 时间窗（start/end）
- 影响范围（cluster/az/worker/client）
- TopN 错误码分布
- 分段时延 Top3（见 §4）
- 初判责任层级（L0（集成方）/L1（平台与网络）/L2（三方件）/L3（数据系统））
- 建议下一跳排查动作

---

## 3. 错误码设计整改（重点：去混叠）

## A. 现状中不合理点

- `1002` 语义过宽：网络不可达、服务未就绪、网关内部 unavailable 都落同码。
- `5(K_RUNTIME_ERROR)` 泛化：大量内部错误复用，缺少模块级可观测性。
- message 文本不稳定：同类错误文案不统一，影响检索和规则命中。

## B. 整改原则

- **主码稳定**：兼容现网，不轻易改主码。
- **子码细化**：新增 `sub_code`（字符串或整数）表达细分类别。
- **结构化优先**：机器先看字段，文本仅做补充说明。

## C. 建议新增子码（示例）

- `1002.NET_UNREACHABLE`
- `1002.PEER_NOT_LISTENING`
- `1002.GATEWAY_UNAVAILABLE`
- `1002.HEARTBEAT_TIMEOUT`
- `5.MMAP_LOOKUP_FAILED`
- `5.FD_TRANSFER_FAILED`
- `5.URMA_PAYLOAD_OVERFLOW`

## D. 改造位置（首批）

- SDK 侧：
  - `client_worker_common_api.cpp`
  - `client_worker_remote_api.cpp`
  - `client_worker_base_api.cpp`
- Worker 侧：
  - `worker_service_impl.cpp`
  - `worker_oc_service_get_impl.cpp`
  - `worker_oc_service_multi_publish_impl.cpp`
- 公共层：
  - `status.h`（错误码注释/契约）
  - 错误码上报与日志宏（统一输出 `code/sub_code/stage/trace_id`）

## 4. 分段时延日志方案（你提到的“分段时间日志”）

目标：对每次请求输出“阶段耗时分解”，自动判断慢点和断点。

## A. 分段定义（读路径建议）

- `S1_CLIENT_PREPARE_MS`：SDK 入参校验、路由准备、缓冲准备
- `S2_CLIENT_TO_WORKER_RPC_MS`：客户端到入口 Worker RPC
- `S3_WORKER_META_MS`：入口 Worker 元数据查询/路由计算
- `S4_WORKER_REMOTE_FETCH_MS`：跨 Worker 拉取（含 UB/TCP 数据面）
- `S5_WORKER_RESP_BUILD_MS`：入口 Worker 回包构建
- `S6_CLIENT_MMAP_COPY_MS`：客户端 mmap/组包/返回用户态
- `TOTAL_MS`

## B. 日志字段契约（结构化）

建议统一 JSON 字段：
- `trace_id`
- `request_id`
- `api`（Get/Set/MSet/Init/RegisterClient）
- `status_code`
- `sub_code`
- `stage`
- `stage_cost_ms`
- `total_cost_ms`
- `worker_id` / `peer_worker_id`
- `transport`（TCP/UB/SHM）

## C. 示例（单条请求的阶段日志）

```json
{
  "trace_id": "4f8c...",
  "request_id": "req-1024",
  "api": "Get",
  "status_code": 1002,
  "sub_code": "1002.NET_UNREACHABLE",
  "transport": "TCP",
  "stages": {
    "S1_CLIENT_PREPARE_MS": 1,
    "S2_CLIENT_TO_WORKER_RPC_MS": 83,
    "S3_WORKER_META_MS": 0,
    "S4_WORKER_REMOTE_FETCH_MS": 0,
    "S5_WORKER_RESP_BUILD_MS": 0,
    "S6_CLIENT_MMAP_COPY_MS": 0,
    "TOTAL_MS": 84
  }
}
```

## D. 自动判断规则（第一版）

- `S2` 明显高且 Worker 无入口日志 -> 偏 L1（平台与网络）
- `S3` 高 -> 偏 L3（数据系统）元数据链
- `S4` 高 + `1006` 升高 -> UB/跨 Worker 链路优先
- `S6` 高 + mmap/fd 异常 -> L1（平台与网络）资源或 L3（数据系统）SHM 实现

---

## 5. 具体整改清单（按阶段）

## Phase 1（1~2 周）：可观测性补齐

- 定义统一日志字段契约（SDK + Worker）
- 补充读/写/Init 的分段耗时打点
- 告警规则按“业务层 + 错误桶 + 前置信号”上线第一版

**验收标准**
- 任意故障 1 分钟内可看到告警
- 3 分钟内可输出“责任层级初判 + Top 慢段”

## Phase 2（2~3 周）：错误码子码化

- 引入 `sub_code`，先覆盖 `1002` 与 `5`
- 统一 message 模板，减少同义多文案
- 文档同步：`sdk-init`、`kv读/写`、`worker启动` 增加子码映射表

**验收标准**
- `1002` 场景可至少拆成 3 类并可检索统计
- 误判率（L1 vs L3）较当前下降 30%

## Phase 3（持续）：演练闭环

- 按 TCP/UB/etcd/磁盘/迁移构造故障注入用例
- 每月演练并固化 Runbook
- 复盘输出“误判原因 -> 规则修订项”

**验收标准**
- 5 分钟内给出恢复动作
- 复盘材料可直接回放到单请求级证据链

---

## 6. 建议先做的三件事（本周）

1. 在 `Get/Set/Init` 先打通 `trace_id + stage_cost_ms + sub_code` 三字段。
2. 先给 `1002` 拆 3 个子码，不改主码，避免兼容风险。
3. 上线一条组合告警：`1002占比 + Worker无入口日志 + S2超时段升高`，用于快速区分网络 vs 数据系统。
