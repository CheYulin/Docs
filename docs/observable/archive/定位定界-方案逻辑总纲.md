# 定位定界方案逻辑总纲（非 PPT 版）

本文将现有两份素材 `定位定界-ppt素材-团队视角版.md`、`定位定界-ppt素材-证据版.md` 收敛为一份可执行的方案逻辑，目标是：
- 明确责任边界（自证清白）
- 提升故障与性能劣化的及时感知
- 形成“观测 -> 定界 -> 恢复 -> 复盘”的闭环

适用部署：64 节点 DataWorker，单节点 8 个推理实例，etcd 5 节点（1 主 4 备）。

---

## 1. 核心目标与原则

## 1.1 核心目标
- **自证清白**：快速判定是否为数据系统责任，若是则定位到具体链路和模块；若不是则给出依赖侧证据。
- **及时感知**：不仅感知报错，还要提前感知 QoS 劣化（尤其 P99）。
- **可恢复**：故障发生后有明确的自动/人工恢复路径，不靠“经验猜”。

## 1.2 原则
- **先分层，再分责**：先按链路层（TCP/UB/etcd/存储/资源）分桶，再做团队责任判定。
- **证据驱动**：必须同时使用状态码、日志、时序/Trace、分段时延，不以单点证据定责。
- **规格先行**：对依赖方定义明确接口与时延规格，违规可定界。
- **统一时间窗**：所有判断以同一时间段对齐（客户端、Worker、依赖侧）。

---

## 2. 系统与链路模型（定界基础）

## 2.1 业务链路
- Client <-> Worker：
  - SHM：本地免拷贝
  - TCP：请求与控制面
  - UB（URMA）：远端数据路径或本地异常后的替代路径
- Worker <-> Worker：
  - UB：数据搬移
  - TCP：元数据与拉取请求
- Worker <-> etcd：租约心跳、集群信息
- Worker <-> 二级存储：异步持久化与恢复

## 2.2 责任层级
- **L0（集成方）**：参数、超时、重试策略、本地缓存策略、部署变更。
- **L1（平台与网络）**：socket、重传、网络质量、调度、mmap/signal、FD 资源。
- **L2（三方件）**：etcd、二级存储、外部控制面能力。
- **L3（数据系统）**：SDK/Worker 逻辑、重试降级、切流、状态机、模块实现。

---

## 3. 观测与告警闭环

## 3.1 必备观测数据
- **指标**（Grafana）：`success_rate`、读/写 `p99`、错误码分布、资源指标（CPU/mem/fd/queue/threadpool）。
- **日志**：接口日志、资源日志、第三方访问日志（etcd/存储）、运行错误日志。
- **时序/Trace**：请求级 trace_id，串联客户端与 Worker 内部关键路径。

## 3.2 告警策略（建议）
- **R1 SLI 告警（必选）**：读/写 p99、success_rate。
- **R2 UB 专项**：1004/1006/1008、UB 分段超规格。
- **R3 TCP/RPC 专项**：1002/1001/23、重传和连接异常。
- **R4 etcd 专项**：25/14 + etcd 成功率下降。
- **R5 资源/文件接口**：6/13/18 + 资源打满或 IO 失败。
- 分级：P1（全局）、P2（局部）、P3（抖动观察）。

说明：Grafana Alerting 支持告警触发与恢复自动消警。

---

## 4. 统一定界流程（五步）

1. **判影响面**：先看 success_rate / p99 是全局还是局部（节点、AZ、单 Worker）。
2. **码桶分流**：
   - TCP/RPC：1000/1001/1002/23/19
   - UB：1004/1006/1008
   - etcd：25/14
   - 资源/文件：6/13/18/7
   - 生命周期：31/32/22
3. **分段耗时定位**：看慢在哪一跳（见第 6 节）。
4. **时间窗对齐日志**：客户端 + 入口 Worker + 对端 Worker + 依赖侧。
5. **输出结论**：责任团队、数据系统内模块、恢复动作、后续改进项。

## 5. 场景化分析框架（你要求的 6 个场景）

## 5.1 SDK Init
- 重点看：`K_INVALID`、`1002`、`23`、fd 交换与 mmap。
- 常见分界：
  - `Connect reset/Network unreachable` -> L1（平台与网络）
  - `Authenticate failed` -> L0（集成方）+L3（数据系统）
  - `GetClientFd/mmap` 异常 -> L1（平台与网络）+L3（数据系统）
- 恢复：重试 -> 切备用 Worker -> 失败隔离。

## 5.2 Worker Init
- 重点看：etcd 初始化链、cluster manager、UDS bind、服务装配。
- 常见分界：
  - etcd 不可用/无主 -> L2（三方件）
  - UDS 路径权限/端口冲突 -> L1（平台与网络）
  - 内部初始化链失败 -> L3（数据系统）
- 恢复：修依赖后重启 Worker，自恢复。

## 5.3 Worker 扩缩容
- 重点码：31（退出态）、32（迁移中）。
- 常见分界：
  - 31 大量出现：生命周期行为，不等于故障
  - 32 长时间不收敛：L2（三方件）+L3（数据系统）
- 恢复：客户端退避切流 + 控制面恢复。

## 5.4 KV 本地读（入口命中）
- 重点路径：入口 RPC + 返回映射（SHM/fd/mmap）。
- 常见问题：局部资源打满、fd/mmap 失败、入口 Worker 压力。
- 分界：资源证据指向 L1（平台与网络）；逻辑/状态机异常指向 L3（数据系统）。

## 5.5 KV 远端读（跨 Worker）
- 重点路径：TCP 元数据/请求 + UB 写回 + 回包。
- 常见问题：
  - 1002：TCP 控制面问题
  - 1006：URMA 需重连
- 恢复：UB 失败降级 TCP；TCP 继续失败执行隔离策略。

## 5.6 KV 写
- 重点码：32、1002、6/13/7、2003（写回压力）。
- 常见分界：
  - 落盘/队列问题 -> L1（平台与网络）+L3（数据系统）
  - 控制面迁移问题 -> L2（三方件）+L3（数据系统）
- 恢复：异步降级、限流退避、修复后回升可靠性等级。

## 6. P99 分段方案（性能定界核心）

对一次请求做统一分段并各自定义规格（按小/中/大对象分档）：
1. `Client -> Worker1` RPC(TCP)
2. `Worker1 -> Worker2` RPC(TCP, 元数据)
3. `Worker1 -> Worker3` RPC(TCP, 拉取请求)
4. `Worker3 -> Worker1` URMA Write
5. `Worker1 -> Client` 返回（SHM/TCP）

定界规则：
- 若单段超规格且重复出现 -> 该段责任域优先。
- 若 UB 段异常同时 TCP 正常 -> UMDK/UB 路径优先。
- 若 UB 降级后 TCP 也异常 -> 网络/OS 或节点系统性问题优先。

---

## 7. 各团队接口契约（对外要求）

## 7.1 UMDK/容器网络团队
- URMA 建链成功语义与完成判定必须明确。
- URMA Write + Completion 的一致性与时延上限必须可验证。
- 提供管理面诊断能力（端点连通、错误原因）。

## 7.2 OS/容器网络团队
- TCP socket 读写时延规格、重传率上限、恢复时间上限。
- 提供故障窗口可观测证据（重传、RST、link down、调度延迟）。
- mmap/SIGBUS 场景提供可复盘信息（信号与地址映射证据）。

## 7.3 中台（etcd/存储）
- etcd：可用性、选主时间、watch 延迟、错误可观测性。
- 存储：pread/pwrite 延迟与错误率、队列容量与恢复机制。

---

## 8. 恢复策略总览

- **UB 异常**：优先切 TCP；若反复失败进入隔离策略。
- **TCP 异常**：重试/退避；持续失败按策略重启/摘除。
- **etcd 异常**：告警中台恢复；数据系统做降级与自动收敛。
- **二级存储异常**：保持业务不中断，可靠性等级降级并告警。
- **资源异常**：SDK/Worker 触发告警并重启，运维保障资源恢复。

---

## 9. 数据系统“自证清白”输出模板

每次事件输出统一四段：
1. **事实**：时间窗、影响面、SLI 变化、主要错误码。
2. **证据**：日志片段、Trace 片段、分段时延、依赖侧状态。
3. **结论**：责任层级（L0（集成方）/L1（平台与网络）/L2（三方件）/L3（数据系统））+ 数据系统内模块（若 L3（数据系统））。
4. **动作**：已执行恢复、待依赖方动作、预防改进项。
