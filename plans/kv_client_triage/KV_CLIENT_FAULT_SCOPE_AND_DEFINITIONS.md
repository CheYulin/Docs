# KVC 故障：定义、处理边界、URMA/OS 指标分解与客户配合

本文与 [KV_CLIENT_FAULT_TRIAGE_TREE.md](./KV_CLIENT_FAULT_TRIAGE_TREE.md)、[cases.md](./cases.md)、[KV_CLIENT_E2E_FRAMEWORK_AND_TRIAGE.md](./KV_CLIENT_E2E_FRAMEWORK_AND_TRIAGE.md) 配套，补齐三件事：

1. **故障处理范围**：哪些能靠客户端/SDK 文档定界、哪些是**硬约束**、**必须客户或基础设施侧配合**；
2. **无法在本轮处理或无法定界时怎么操作**（止损、升级、观察、Gap 记录）；
3. **故障如何定义**，以及**分解到 URMA（UB 链路）与 OS/主机**侧时各看什么指标（**示例名需与现网监控/运维平台对齐**，非产品强绑定字段）；
4. **数据系统与业务系统均无法闭环时谁兜底、怎么做**（联合升级、事件 Owner、业务连续性）。

---

## 1. 故障如何定义（分层，避免混用口径）


| 层级         | 定义要点                                                                     | 典型用途                                 |
| ---------- | ------------------------------------------------------------------------ | ------------------------------------ |
| **业务/SLI** | 客户监控上的 **成功率、TP99、E2E 失败率** 相对 **阈值/SLA** 的违背（含持续时间、影响面）                 | 告警、是否构成「事件」                          |
| **客户端可观测** | SDK `**StatusCode`**、[access log](./KV_CLIENT_TRIAGE_PLAYBOOK.md) 行、业务日志 | 第一层分类：RPC / URMA / 未就绪 / NOT_FOUND 等 |
| **系统根因**   | [cases.md 故障模式](./cases.md) 中的大类：主机/OS、TCP、UB、Worker、etcd…               | 定界到责任方后，由 **L4/L5** 证据确认             |


**注意**：**同一 SLI 异常** 不等于 **已定位根因**；**同一 StatusCode**（如 `1001`）可能对应 **TCP 与 UB 多类根因**（见 [KV_CLIENT_CLIENT_PERSPECTIVE_REPORTS.md](./KV_CLIENT_CLIENT_PERSPECTIVE_REPORTS.md) 报告一）。

## 2. 分解：URMA（UB 链路）与 OS/主机 — 各看什么指标

下表用于 **L5 定界**：客户端已出现 `K_URMA_`* 或断连/超时类码时，在基础设施侧 **分别** 查 UB 与主机，避免只盯一侧。

### 2.1 URMA / UB 侧（用户态 RDMA 路径、Jetty、平面）


| 观测意图   | 指标/事件类型（示例名，按现网落地）                              | 与客户端/SDK 的对应关系（提示）                                                                                              |
| ------ | ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| 链路可用性  | UB **端口** up/down、闪断次数；**平面**切换、降 **lane**      | `K_URMA_`*、TP99 长尾（降 lane 可能「只伤长尾」）；见 cases **UB 链路故障**                                                         |
| 时序与窗口  | 故障检测与平面切换 **~128ms 量级**、切换完成时间                  | 与 **客户 timeout（如 20ms）** 叠加时现象复杂，见 cases **时间指标**、[REMOTE §4.5](./REMOTE_META_VS_DATA_FAILURE_OBSERVABILITY.md) |
| 数据面写/读 | URMA **错误码**（如环境定义的 Error 09 等）、**Jetty** 会话/重连 | Predictor/接口日志中的 URMA Error；`K_URMA_NEED_CONNECT`、`K_URMA_TRY_AGAIN`                                            |
| 容量与拥塞  | UB **带宽利用率**、队列深度、重试（若平台暴露）                     | 成功率与时延同时劣化；与 **TCP 回退** 路径交叉时需结合 Worker 日志                                                                      |


### 2.2 OS / 主机侧（内核、进程、带外）


| 观测意图     | 指标/事件类型（示例名，按现网落地）                      | 与客户端/SDK 的对应关系（提示）                                     |
| -------- | --------------------------------------- | ------------------------------------------------------ |
| 生命周期     | **重启**、**Panic**、**BMC 上下电**、维护窗口       | 多实例同时异常；`K_RPC_UNAVAILABLE`、连接失败；见 cases **主机/OS** 1–3 |
| 资源       | **CPU**、**内存**、**OOM**、**磁盘** IO、**fd** | 排队导致 TP99 升；未必是 URMA 码                                 |
| 时间       | **NTP 跳变**、大时钟偏移                        | 元数据/租约类问题（依架构）；与 **128ms** 等窗口叠加分析                     |
| 网络栈（主机内） | 网卡驱动统计、软中断（与 **L5 TCP** 协同）             | `1001`/`1002` 与 UB 区分困难时需 **TCP 侧** 指标 + UB 侧对照        |


**说明**：**URMA 问题** 与 **OS/主机问题** 可能同时存在（例如主机高负载拖慢用户态 URMA）；定界时 **两边都看**，用时间线与 blast radius 收敛。

## 3. 故障处理范围：能做什么 vs 做不了（约束）

### 3.1 本目录分析/SDK 视角**通常能覆盖**的

- 从 **业务监控 → SDK Status → client access log** 做 **第一层** 分类与 **DryRun 顺序** 收窄（见 [cases DryRun](./cases.md)）。
- 结合 **cases 可靠性表** 说明 **设计意图内的重试、切流、TCP 回退** 与 **时间粗算**（2/N 等）。
- **责任方粗分**（应用与集成 / 客户运维 / 网络与机房 / KVC 平台，见 [排查树](./KV_CLIENT_FAULT_TRIAGE_TREE.md)）：便于沟通，**非**最终责任认定。

### 3.2 **硬约束**（识别「做不了」或「仅靠文档不够」）


| 约束                | 含义                                                                                    | 对客户/现场的含义                                                                                                       |
| ----------------- | ------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| **同码多因**          | `K_RPC_DEADLINE_EXCEEDED` / `K_RPC_UNAVAILABLE` 等 **无法单独** 区分 TCP 网卡、UB 交换机、对端 Worker | 必须 **L5 指标或抓包** 或 **厂商侧日志**                                                                                     |
| **远端多跳不可见**       | 客户端易见 **入口 Worker**；**数据面/元数据** 远端节点 **无权限或无日志集中** 时                                  | 依赖 **Trace + 定点 grep**、或 **协议增强（GetRsp 带 HostPort）**；见 [REMOTE](./REMOTE_META_VS_DATA_FAILURE_OBSERVABILITY.md) |
| **元数据失败被掩盖**      | `GetObjMetaInfo` 失败可能仅 WARNING，仍继续 `Get`                                              | 业务 **无法** 仅从最终 Status 区分 meta/data；需 **指标分阶段/协议**（REMOTE）                                                       |
| **短超时 vs UB 检测窗** | 例如 **20ms** 预算与 **~128ms** UB 感知 **不同量级**                                             | 可能出现 **超时先于平台自愈**；产品语义与是否 TCP 回退需对齐（REMOTE §4.5）                                                                |
| **鉴权**            | 本 triage **不包含** Token/AKSK                                                           | 鉴权类问题走 **独立流程**                                                                                                 |


### 3.3 **需要客户配合**时（必须点出）


| 场景       | 客户需配合的内容                                      | 原因              |
| -------- | --------------------------------------------- | --------------- |
| L1 定界    | 提供 **负载均衡器/业务监控** 上实例维度、时间窗                   | SDK 看不到全网关侧     |
| L5 网络/UB | **机房/网络团队** 开放端口指标、抓包窗口、维护公告                  | 厂商侧通常不直连客户交换机   |
| 日志与工单    | **业务实例列表**、**TraceID**、**复现时间段**、是否 **发布/变更** | 缩短盲 grep 与误伤责任方 |
| 集成与容量    | **Init 顺序**、**timeout/重试**、**扩容**             | 见 E2E 六项之 3、4   |
| 止损决策     | **是否启用降级路径**（不读 KV 等）                         | 业务策略，非 SDK 默认   |

## 4. 针对「无法处理」或「暂无法定界」时怎么操作


| 阶段          | 建议动作                                                                                                    | 说明                          |
| ----------- | ------------------------------------------------------------------------------------------------------- | --------------------------- |
| **止损**      | 限流、降级非关键读、临时调大 timeout（**权衡尾延迟**）、切流/只读策略（若业务支持）                                                        | 降低收入影响；不替代根因修复              |
| **升级工单**    | 打包：**现象**（SLI）、**范围**（全局/批次）、**时间线**、**Status 分布**、**access log 抽样**、**已锁定 Worker IP:Port**、**TraceID** | 满足「可复现、可缩小」即可，不必等全集群 grep 完 |
| **设计意图内自愈** | 对照 cases **隔离/切流/重试** 窗口，**观察是否恢复**                                                                     | 避免在自愈窗口内过度变更                |
| **确认 Gap**  | 若属于 **§3.2** 约束（如 meta/data 不可分、缺 HostPort 回传），记为 **已知可观测性 Gap**，跟 **版本/需求**                            | 避免重复无效排查                    |


**无法处理** 在此指：**在客户可见证据与权限下仍不能收敛根因** —— 应 **升级** 而非无限本地 grep；**无法恢复** 指 **违背 SLA 且超出设计自愈** —— **止损 + 工单 + 变更回滚** 并行。

## 5. 数据系统与业务系统都搞不定时：谁兜底、怎么做

本节回答：**平台/数据系统（KVC、Worker、SDK 等）与业务应用侧各自排查后仍无法定责或无法恢复时**，由谁抬升、以什么机制闭环 —— **不是**再增加一层「技术细节」，而是 **组织与流程上的兜底**（具体角色名以各客户 **运维制度 / 合同** 为准）。

### 5.1 「兜底」指什么

- **不是**自动出现某个「超级团队」在后台修好；而是 **明确升级路径**、**单一事件 Owner**、**业务连续性优先**。
- **目标**：在根因未完全厘清前，先 **控制影响**（止损）；同时 **避免双方向推诿**（数据侧说应用、应用侧说链路）。

### 5.2 典型兜底责任方（按场景组合，可重叠）


| 角色（通用称呼）                        | 何时进入兜底                           | 典型职责                                               |
| ------------------------------- | -------------------------------- | -------------------------------------------------- |
| **客户侧事件/值班负责人**（或 SRE、运维指挥）     | SLI 持续违背、影响收入/承诺窗口               | **统一对外口径**、拍板 **是否全站降级/限流**、拉齐各团队时间线               |
| **客户侧业务负责人 / 产品**               | 止损策略涉及 **业务语义**（哪些读可砍、是否只读）      | 与 **数据系统 SLA**、**合同** 对齐；批准 **风险接受**（短期带损运行）       |
| **数据系统 / 平台厂商接口**（技术支持二线、产品、架构） | 已提供日志与复现，仍 **无法解释行为** 或 **疑似缺陷** | **深度日志/版本**、**已知问题**、**补丁/热修路径**；协调研发              |
| **基础设施责权方**（机房、网络、运营商、UB 硬件维保）  | L5 指向 **非单产品可解**（交换机、线路、多租户争抢）   | **抓包授权**、**维护窗口**、**运营商工单**；客户侧常需 **合同内接口人** 打开    |
| **联合升级 / 管理层**                  | 跨团队 **72h 级**仍无根因，或 **合同/商务** 介入 | **资源加码**、**厂商高层升级**、**外部专家**；非首选，但需 **明确触发条件**（见下） |


**要点**：兜底 **通常不是「一个人」**；而是 **客户指定一个事件 Owner + 厂商侧对口升级**，其余角色 **按证据入场**。

### 5.3 怎么做（推荐动作顺序）

1. **冻结无结论变更**：在未统一结论前，避免业务与平台 **同时** 做互不感知的变更（除非明确回滚）。
2. **单页时间线（Timeline）**：从 **首次告警 → 发布/网络事件 → 双方已做动作** 对齐到分钟级，附 **Status 主簇、影响面**（见 §4 工单包）。
3. **联合定界会议（War room）**：数据系统、业务、网络（若已卷入）**同一会议** 过证据；输出 **「下一跳只证伪一个假设」** 的动作（谁执行、何时交结果）。
4. **业务连续性兜底**（与技术根因并行）：
  - **已文档化**的降级（不读 KV、静态兜底、读副本等）；  
  - 若合同允许，**临时调 SLA/限流** 并 **书面记录风险接受**。
5. **仍无根因时**：
  - **产品侧**：开 **缺陷/Gap**（可观测性、协议、超时语义），跟 **版本**；  
  - **客户侧**：**风险运行** 策略 + **复盘**（RFO 类）；  
  - **商务/合同**：若涉及 **赔偿/服务信用**，走 **客户既定流程**（本文不展开条款）。

### 5.4 与本目录其它条目的关系

- **§3.2 硬约束**（同码多因、远端不可见等）是 **容易进入 §5 的典型技术原因** —— 不是「没人管」，而是 **必须升级权限与联合证据**。
- **§4 止损/工单** 是 **进入 §5 之前** 应做完的；§5 是 **仍无解** 时的 **组织兜底**，与 **技术兜底**（降级、绕行）并行。
